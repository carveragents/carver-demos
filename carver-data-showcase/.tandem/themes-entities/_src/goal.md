Themes & Entities — a new Gallery tab + matching deck slide for the Carver Annotation Data Showcase that turns the annotations' `tags` and `entities` fields into corpus-wide stats, using a ONE-TIME OpenAI Batch enrichment to type & de-duplicate entities (tags stay deterministic frequency stats).

## Overall brief (carried into every stage)

**What this adds.** The showcase today surfaces scores, jurisdictions, regulators, update-types, richness and a single-record drill-down — but two of the richest annotation fields never get an aggregate view: `tags` (free-text thematic labels) and `entities` (named organisations / people / places). This feature adds a dedicated **"Themes & Entities"** surface (a new Gallery tab + one new deck slide) that shows:
- **Entities, typed:** a 6-bucket breakdown of what kinds of entities the corpus references, plus a "most-referenced bodies" leaderboard with aliases merged.
- **Tags, as themes:** a top-N themes bar + diversity headline (no LLM — pure frequency).

**The data (verified against the full 211,489-record snapshot).** Both fields live as **flat lists of plain strings** under `output_data.metadata.tags` and `output_data.metadata.entities`. Coverage is high — tags present on 88.5% of records, entities on 92.5%. The corpus holds **234,002 distinct tags** (1.54M mentions, median 8/record) and **281,180 distinct entities** (900K mentions, median 3/record), both brutally long-tailed (~60-70% of distinct values appear once). Entities are **NOT pre-typed** (0 of 900K instances carry a type; they are raw strings like `["Christopher P. Buttigieg", "EBA", "Malta Financial Services Authority", "MFSA"]`). The named-entity extraction is **regulator-heavy** — the top entities are a who's-who of global regulators (European Commission, ECB, FTC, ESMA, SEC, FDIC, EBA, EMA, FDA, IMF, FCA…), with companies (Mastercard, CIBanco), people, and even places (Bolivia, Croatia) in the tail.

**Why an LLM now.** A pre-LLM heuristic pass proved a clean person/company/regulator split is NOT reliably derivable by rule (e.g. "Deutsche Bundesbank" trips a person-name regex; ~34% stay unclassifiable). The project's earlier v1 no-LLM stance is therefore **deliberately lifted for THIS feature**: entity typing + alias resolution use a one-time OpenAI Batch job. Cost is ~$1 for all 281K distinct entities, cached forever, so it never runs at app load.

**Locked decisions (do not relitigate):**
1. **Scope = entities get the LLM; tags do NOT.** Entities are typed + de-duplicated by the model. Tags are shown as deterministic frequency stats only (top-N bar + diversity tiles). A tag-clustering pass is explicitly a possible future follow-up, OUT of scope here.
2. **Entity taxonomy = 6 fixed, mutually-exclusive buckets:** `Regulator / Supervisor` (incl. central banks) · `Government body` (ministries, legislatures, courts, law-enforcement) · `International body` (intergovernmental & standard-setting) · `Company` · `Person` · `Other` (places + unclassifiable). Unclassifiable → `Other`.
3. **Alias merge.** Each classification returns `{type, canonical_name}`; the leaderboard counts by `canonical_name` so all variants of one body (SEC / Securities and Exchange Commission / U.S. Securities and Exchange Commission) collapse into one bar with the combined total. A cheap DETERMINISTIC cleanup pass (strip leading `U.S. `, unify punctuation/case/whitespace) runs at rollup time. The 6-bucket breakdown is unaffected by merging.
4. **Transport = OpenAI Batch API for the full run.** ONE batch job: a single `.jsonl` of ~5,600 request lines, each line classifying ~50 entities (`CHUNK_SIZE`, tunable). Async submit → poll → fetch → cache. A sync `--sample N` mode (plain Chat Completions, a few hundred entities) exists for prompt iteration. The "50 per request" is sharding INSIDE the one batch (output-token safety + retryable failures), not a separate batch.
5. **Model = `gpt-4o-mini`** (configurable constant). Structured/JSON output. ~$1 one-time for the full distinct set.
6. **Stats are aggregate / full-corpus.** The lean parquet stores only `n_tags`/`n_entities` counts, so this surface shows corpus-wide aggregates and **does NOT react to the sidebar filters** the other tabs honour. A caption states "across the full corpus." Per-record entity filtering (e.g. entities for one jurisdiction) needs per-record entity storage + join and is **deferred (YAGNI for v1)**.
7. **Surfaces = Gallery tab + deck slide, sharing `charts.py` builders.** A new "Themes & Entities" tab becomes the Gallery's 9th tab; the deck grows 8→9 slides using the SAME pure `df→go.Figure` builders, so gallery and deck cannot drift.
8. **Stack/secrets:** Python 3.12 `.venv/`; add `openai` to requirements. `OPENAI_API_KEY` already lives in the git-ignored `.env` and is read **only inside the enrichment tools**, never in the app or chart code. The big `annotations.jsonl` and all derived artifacts stay git-ignored (regenerated by tools), exactly like the existing parquet.

**Hard constraints inherited from the project (do not violate):**
- **This is Gallery + deck ONLY. Do NOT touch the internal Data-Quality Cockpit** or its data path.
- **Categories stay internal** — never surface topic categories on this external tab/slide. (Tags & entities ARE annotation content and are fine to show externally.)
- **No commit.** All work stays uncommitted; the user merges via their own flux workflow.
- **TDD + right-sized models + run python-code-reviewer after code changes**, per the repo's CLAUDE.md.

**North star:** make the corpus's *breadth of who and what it talks about* tangible — "thousands of regulators, companies, and people across 234K themes" — with one honest, cached, model-typed entity breakdown and a clean themes view, without ever calling a model at render time or breaking the cockpit.
