# Site Build

> **Stage 0 deliverable.** Defines the tech stack, repo layout, build pipeline, and deploy target. After Stage 0, the site shell + landing page deploys; each subsequent stage adds scenes.

---

## 1. Tech Stack

Optimized for: minimum machinery, fast build, browser-only runtime, easy "view source" inspection, easy edit by anyone who knows HTML.

| Concern | Choice | Rationale |
|---|---|---|
| Language | Python 3.10 (matches repo target) | Already pinned (see [`../development.md`](../development.md)). |
| Templating | Jinja2 | Standard, fast, familiar; same template flavor a future production build would use. |
| CSS | Tailwind CSS via CDN (`<script src="https://cdn.tailwindcss.com">`) | No build step; production-ready aesthetic; trivially editable. |
| JS framework | None. Vanilla JS + small Alpine.js sprinkles where genuinely needed (`<script defer src="https://unpkg.com/alpinejs">`). | Avoids bundler tax for a static site. |
| Charts | Apache ECharts via CDN (`<script src="https://cdn.jsdelivr.net/npm/echarts">`) | World map + heat-map + sparklines all in one library; renders from JSON. |
| Icons | Lucide icons via CDN | Clean, professional, no asset pipeline. |
| Fonts | System UI stack + one webfont (Inter via Google Fonts) | One network dep, fast load, modern look. |
| Markdown rendering (for ticket comment bodies and walkthrough copy if reused) | `markdown-it-py` at build time → HTML | Build-time render avoids runtime JS. |
| Carver data pull | [`carver-feeds-sdk`](https://github.com/carveragents/carver-feeds-sdk) (PyPI) | Official SDK; topic/feed/entry queries via fluent chainable filters. Auth via `CARVER_API_KEY`. |
| Platform API pull | `httpx` direct calls to Kalshi (`external-api.kalshi.com/trade-api/v2`) + Polymarket (`gamma-api.polymarket.com`, `clob.polymarket.com`). | Public, no-auth read endpoints. SDK overhead unnecessary for ~50 markets per platform. |
| Local dev server | `python -m http.server` against `site/` | Built-in; sufficient for static. |
| Deploy | GitHub Pages (default) | Free, public, version-pinned to a branch. Alternatives in § 6. |

**No Node, no npm, no webpack, no bundler.** Anyone with Python 3.10 can build and serve.

## 3. Build Pipeline

Three stages, each a single command:

```
   pull_carver.py     →     generate_slices.py     →     generate.py
       (occasional)              (after data edits)             (every build)

   data/carver-events.json  →  build/page_data/**.json  →  site/**/*.html
```

### 3.1 `pull_carver.py`

Reads `../carver-dags/workflows/entry_annotation/` (approach decided in Stage 0 per data-prep doc DP1), applies the filter from data-prep § 2.1, writes `data/carver-events.json` + `data/carver-pull-manifest.json`. Run rarely — only when refreshing the underlying corpus.

### 3.2 `generate_slices.py`

Pure-Python, no network. Reads `data/**` (Carver JSON + YAMLs), computes per-page slices, writes `build/page_data/**`. Idempotent. Spec in [`10-data-prep.md`](10-data-prep.md) § 5.

### 3.3 `generate.py`

For each template in `build/templates/`, loads any required slice from `build/page_data/`, renders via Jinja2, writes the output to the corresponding path under `site/`. Copies `build/static/` verbatim into `site/static/`. Produces a sitemap.

```python
# Pseudocode of the render loop
for tpl in walk('build/templates/'):
    out_path = tpl.replace('build/templates/', 'site/').replace('.html', '.html')
    ctx = load_slice(tpl)   # convention: templates/alpha/inbox.html → page_data/alpha/inbox.json
    site_render(tpl, ctx, out_path)
copy_tree('build/static/', 'site/static/')
write_sitemap()
```

Loop is data-driven from the templates dir — adding a new page is "drop a template + a slice."

### 3.4 Makefile

```makefile
.PHONY: pull slice build serve clean

pull:
	python build/pull_carver.py

slice:
	python build/generate_slices.py

build: slice
	python build/generate.py

serve: build
	cd site && python -m http.server 8000

clean:
	rm -rf build/page_data site
```

`make build` is the daily command. `make pull` is rare.

## 4. Templating Conventions

### 4.1 Base layout

`build/templates/base.html` provides:
- `<head>`: title block, meta tags (description, OpenGraph, Twitter card), CDN scripts (Tailwind, ECharts, Alpine, Lucide, Inter font), `site/static/css/site.css`.
- Top navigation bar: Pred-Oracle wordmark, breadcrumb / scene indicator (Landing / α / γ / β / Close), "Demo" badge.
- Main content slot: `{% block content %}`.
- Footer: small print — *"Pred-Oracle demo built on Carver regulatory annotations. Synthetic platform context labelled where applicable. Real persons / events linked to public sources."*
- A persistent "next-scene" CTA pinned bottom-right (where applicable).

### 4.2 Per-page templates

Each extends `base.html` and overrides `content`. Conventionally, the first 100 lines of any template are layout/markup; data is referenced via Jinja vars loaded from the corresponding `page_data/*.json`.

### 4.3 Components

`build/templates/_components/` holds reusable partials:
- `_components/ticket_row.html` — one row of α inbox.
- `_components/event_card.html` — Carver-event summary card used in α ticket detail, γ contract detail, β drilldown.
- `_components/heat_cell.html` — one cell of β heat-map (or α dashboard).
- `_components/jurisdiction_pill.html` — uniformly-styled jurisdiction chip.
- `_components/source_badge.html` — "View primary source" link with regulator favicon.
- `_components/demo_badge.html` — small "demo data" badge for synthetic comment threads / assignees.

Templates use `{% include %}` (or macros for the ones that take arguments).

### 4.4 Styling

Tailwind utility-first. Custom CSS in `site.css` only for:
- `@apply` shortcuts where a class string repeats 5+ times.
- A few brand colors / spacing tokens beyond Tailwind defaults.
- One `print:` ruleset to make the quarterly-report page printable to PDF from a browser.

Color tokens (proposed):
- Brand primary: `#0F172A` (slate-900) for navigation chrome.
- Accent: `#2563EB` (blue-600) for CTAs.
- Urgency scale (used by ticket rows, heat cells): green/yellow/orange/red gradient via Tailwind's `bg-green-50` → `bg-red-100`.

## 6. Deployment

### 6.1 Default: GitHub Pages

- Branch: `gh-pages` (deploy-only; auto-published by GitHub).
- Workflow: GitHub Action that runs `make build` on push to `main`, commits `site/` into `gh-pages`, force-pushes.
- URL: `https://<org>.github.io/pred-oracle/`.
- Trade-offs: public, free, no auth. Acceptable because the data is public (Carver annotates public regulatory content) and the platform-context is sourced from public info.

Action file (`.github/workflows/deploy.yml`) is in scope for Stage 0. Sketch:

```yaml
on:
  push: { branches: [main] }
jobs:
  build-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.10' }
      - run: pip install uv && uv sync
      - run: make build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./site
```

### 6.2 Alternative: gated deploy (if access control is wanted)

- **Vercel** or **Netlify** with a "preview" team / basic-auth env vars.
- Or self-host on Fly.io / Render behind HTTP basic auth.

Defer the decision to Stage 0. Default to GitHub Pages unless someone surfaces a privacy concern about the data.

### 6.3 Local preview

`make serve` exposes the built site on `localhost:8000`. Sufficient for design review and demo-script rehearsal.

## 7. Performance Budget

Per page:
- First contentful paint: <1s on a typical broadband connection.
- Total page weight: <500KB (gzipped), including CDN-loaded Tailwind + ECharts (charts pages exempted at <1MB).
- Lighthouse score: ≥90 for performance, accessibility, SEO.
- No layout shift after load (preload font, fixed-aspect chart containers).

Heat-map page is the most chart-heavy; cap its data slice at ~5000 rows (jurisdiction × week buckets) to keep ECharts render snappy.

---

## 8. Accessibility

- Semantic HTML (`<nav>`, `<main>`, `<article>`, `<aside>`).
- All interactive elements keyboard-navigable.
- Color contrast: at least WCAG AA. Verify the urgency-scale colors against background.
- Alt text on all images and icons (Lucide icons inherit ARIA labels).
- Charts have a "view data" toggle that exposes the underlying numbers as a `<table>`.
