# CER Report Generation — Medical Device Lifecycle Platform

A React demo that combines real post-market surveillance data with live regulatory intelligence from the [Carver Feeds API](https://carveragents.ai) to support Clinical Evaluation Report (CER) generation for cardiovascular medical devices.

Locked to a single demo device: **CardioWatch X1** (510(k) K213456, Class IIb EU / Class II US, markets: US · EU · AU · IN).

---

## Stack

- **Frontend** — Vite + React 18 (functional components, hooks)
- **PMS data** — FDA MAUDE · MHRA · Swissmedic (preprocessed from raw JSON dumps)
- **Regulatory intelligence** — Carver Feeds API (48 medical device topics, 30-day rolling window)
- **Vectorization** — Pure Python TF-IDF + cosine similarity (no ML framework required)

## Quick Start

### 1. Install frontend dependencies

```bash
npm install
npm run dev
```

### 2. Set up Python environment

```bash
python -m venv .venv
source .venv/bin/activate       # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 3. Configure credentials

Create a `.env` file in the project root:

```env
CARVER_API_KEY=your_api_key_here
CARVER_USER_ID=your_user_id_here
CARVER_BASE_URL=https://app.carveragents.ai
```

### 4. Run the regulatory intelligence pipeline

```bash
# Fetch last 30 days of regulatory entries from Carver API
python scripts/fetch_feeds.py

# Vectorize + rank by cardiovascular device relevance
python scripts/vectorize_feeds.py
```

This writes `regulatory_feed.json` and `regulatory_horizon.json` to `public/data/`. The React app picks them up automatically on next load.

**Options:**

```bash
python scripts/fetch_feeds.py --days 60          # wider look-back window
python scripts/vectorize_feeds.py --threshold 0.02 --feed-limit 15
```

## The Four Phases

| Phase | Name | Data Source |
|-------|------|-------------|
| 1 | Intelligence Feed | Carver Feeds (live) · static fallback |
| 2 | Similar Device Tracker | Static predicates · FDA/MHRA/Swissmedic AE data |
| 3 | Regulatory Horizon | Carver Feeds (live) · static fallback |
| 4 | CER Readiness | Static (CER section status, countdown, priority list) |

Phase 1 and Phase 3 automatically switch to live Carver data when `regulatory_feed.json` / `regulatory_horizon.json` are present, and display a "Live · Carver Feeds" timestamp badge. If the files are absent, they fall back silently to curated static data.

## Regulatory Intelligence Pipeline

```
Carver API (/api/v1/feeds/topics)
    │
    ├── Filter: 48 medical device regulatory topics
    │   (FDA, MHRA, TGA, Swissmedic, CDSCO, EMA, BfArM,
    │    Paul-Ehrlich-Institut, ANSM, IMDRF, HALM, …)
    │
    ├── Fetch entries: last 30 days (~1,963 entries per run)
    │
    ├── Fetch AI annotations per topic (~6,369 annotations per run)
    │        → impact score, urgency, summary
    │
    └── vectorize_feeds.py
            │
            ├── TF-IDF over all entries + cardiovascular reference query
            ├── Cosine similarity scoring
            ├── Threshold filter (default 0.03) → ~300 relevant entries
            │
            ├── Forward-looking entries → regulatory_horizon.json (Phase 3)
            └── Actionable entries     → regulatory_feed.json    (Phase 1)
```

## Preprocessing PMS Data (one-time)

The raw medical device surveillance data lives in `medical-device-data/`. To regenerate the filtered JSON assets:

```bash
python scripts/preprocess.py
```

This applies a cardiovascular regex filter and normalises the three source schemas (FDA MAUDE, MHRA, Swissmedic) into a shared slim format, capping US records at 1,000.
